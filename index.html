Locator Chaining & Filtering

Advanced locator operations 
// Filter by text
page.locator('button').filter({ hasText: 'Submit' })

// Filter by another locator
page.locator('div').filter({ has: page.locator('button') })

// Get by index(use sparingly)
page.locator('button').nth(0)
page.locator('button').first()
page.locator('button').last()

// Count locators
await page.locator('button').count()

// Chain locators
page.locator('.product-card').locator('button.add-to-cart')
page.locator('form').getByRole('button', { name: 'Submit' })

// Get all matching elements
const elements = await page.locator('.item').all()
for (const element of elements) {
    await element.click()
}
Avoid using .nth(), .first(), .last() when possible. They create brittle tests. Use more specific locators instead.

Role HTML Elements Example

Locator Best Practices

Priority Order for Locators:
1. getByRole() - Most resilient and accessible
2. getByLabel() - Great for form fields
3. getByTestId() - Stable for automation
4. getByText() - Good for unique text
5. CSS/XPath - Use as last resort
Good vs Bad locators 
// ❌ BAD - Brittle, will break easily
page.locator('div > div > button:nth-child(3)')
page.locator('body > div.container > div.row > button')

// ✅ GOOD - Resilient and meaningful
page.getByRole('button', { name: 'Submit Order' })
page.getByTestId('submit-order-button')
page.locator('button[aria-label="Submit Order"]')

// ❌ BAD - Too generic
page.locator('button').nth(2)
page.locator('.btn').first()

// ✅ GOOD - Specific and stable
page.locator('form[name="checkout"]').getByRole('button', { name: 'Pay Now' })
page.getByTestId('checkout-pay-button')

Locator Strategies

✅ DO:
• Use role-based locators: getByRole('button', { name: 'Submit' })
• Use test IDs for stable automation: getByTestId('submit-btn')
• Use semantic selectors: getByLabel('Email')
• Chain locators for specificity: page.locator('form').getByRole('button')

❌ DON'T:
• Rely on CSS classes: .btn-primary-dark-theme
• Use nth selectors: button:nth-child(3)
• Use XPath unless absolutely necessary
• Use overly specific selectors: div > div > div > button

Auto-Waiting Best Practices

Leverage auto-waiting 
// ✅ GOOD - Auto-waits for element
await page.getByRole('button').click()

// ❌ BAD - Unnecessary explicit wait
await page.waitForSelector('button')
await page.click('button')

// ✅ GOOD - Wait for specific state
await page.getByRole('button').waitFor({ state: 'visible' })

// ✅ GOOD - Wait for network
await Promise.all([
    page.waitForResponse('**/api/data'),
    page.getByRole('button').click()
])

// ❌ BAD - Arbitrary timeout
await page.waitForTimeout(3000) // Flaky!

Test Independence

Each test should be independent:
• Tests should not depend on execution order
• Clean up test data after each test
• Don't share state between tests
• Use fixtures for common setup

Fix flaky tests

test('a11y report', async ({ page }) => {
